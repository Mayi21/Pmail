## 🔴 高危问题（立即修复）

### 1. 敏感数据加密缺失 ⚠️ CRITICAL

**问题描述**:
- 邮件内容（`body_text`, `body_html`, `raw_content`）以明文存储在 D1 数据库
- 附件内容未加密存储在 R2
- 数据库泄露将直接暴露所有用户邮件内容

**风险等级**: 🔴 严重

**影响范围**: 
- 用户隐私完全暴露
- 违反数据保护法规（GDPR、CCPA）
- 品牌声誉受损

**涉及文件**:
- `schema.sql` - 数据库表结构
- `workers/email/src/index.ts` - 邮件存储逻辑
- `workers/api/src/routes/email.ts` - 邮件读取逻辑

**整改要点**:
- 实现 AES-GCM 加密/解密函数
- 在 `workers/api/src/utils/crypto.ts` 添加加密工具
- 存储前加密，读取后解密
- 设置 `DATABASE_ENCRYPTION_KEY` 环境变量
- 数据迁移加密现有邮件

**实施时间**: 立即（1-2周）

---

### 2. JWT 令牌刷新机制缺失 ⚠️ HIGH

**问题描述**:
- 当前仅使用 7 天过期的访问令牌
- 无 Refresh Token 机制
- 令牌被盗后有效期过长

**风险等级**: 🔴 高危

**当前实现**: `workers/api/src/services/jwtKeyManager.ts:150`

**影响范围**:
- 用户体验差（7天后需重新登录）
- 令牌泄露后长期有效
- 无法远程撤销会话

**整改要点**:
- 实现双令牌机制（Access Token 15分钟 + Refresh Token 30天）
- 启用 `sessions` 表存储 Refresh Token
- 添加 `/api/auth/refresh` 端点
- 前端自动刷新令牌逻辑
- Token 轮换机制

**实施时间**: 立即（1周）

---

### 3. 密码重置令牌安全性不足 ⚠️ HIGH

**问题描述**:
- 当前使用 `crypto.randomUUID()` 生成重置令牌（`workers/api/src/routes/auth.ts:318`）
- UUID 可预测性较强，存在暴力破解风险
- 令牌未限制使用次数
- 令牌未绑定 IP 地址

**风险等级**: 🔴 高危

**影响范围**:
- 账户劫持风险
- 暴力破解可能性
- 令牌重放攻击

**整改要点**:
- 使用 64 字符强随机令牌
- 存储令牌的哈希值而非明文
- 实现一次性令牌（使用后立即失效）
- 记录令牌使用的 IP 地址
- 缩短有效期至 1 小时
- 使用后强制登出所有会话

**实施时间**: 立即（3天）

---

### 4. Content Security Policy (CSP) 缺失 ⚠️ HIGH

**问题描述**:
- 当前仅有基本安全头（`workers/api/src/index.ts:104`）
- 未实施严格的 CSP，无法有效防御 XSS 攻击
- 缺少对内联脚本和样式的限制
- 无 frame-ancestors 保护

**风险等级**: 🔴 高危

**影响范围**:
- XSS 攻击风险
- 数据注入攻击
- 点击劫持攻击
- 跨站脚本执行

**整改要点**:
- 在 API 响应头添加严格的 CSP 策略
- 配置 `script-src`, `style-src`, `img-src` 等指令
- 允许 Turnstile 域名（challenges.cloudflare.com）
- 设置 `frame-ancestors 'none'`
- 启用 `upgrade-insecure-requests`
- 逐步移除 `unsafe-inline`，使用 nonce 或 hash

**实施时间**: 立即（1周）

## 🟠 中危问题（近期加强）

### 5. 邮件内容 XSS 防护不足

**问题描述**:
- 显示 HTML 邮件时缺少严格的内容净化
- 恶意邮件可能包含 XSS 攻击代码
- 当前未见 DOMPurify 等净化库的使用

**风险等级**: 🟠 中危

**整改要点**:
- 安装并使用 DOMPurify 库
- 创建安全的邮件内容显示组件
- 严格限制允许的 HTML 标签和属性
- 移除所有脚本和事件处理器
- 沙箱化所有外部链接
- 通过代理加载外部图片

**实施时间**: 2周内

---

### 6. API 输入验证增强

**问题描述**:
- 部分端点可能缺少完整的输入验证
- 需要统一的验证策略
- SQL 注入风险需要持续关注

**风险等级**: 🟠 中危

**整改要点**:
- 创建通用验证 Schema（Zod）
- 增强邮箱、用户名、密码验证规则
- 为所有 API 端点添加完整验证
- 统一错误响应格式
- 验证文件上传参数

**实施时间**: 2-3周

---

### 7. 会话管理增强

**问题描述**:
- `sessions` 表未充分利用
- 缺少设备管理功能
- 无法远程登出或查看活跃会话
- 缺少可疑登录检测

**风险等级**: 🟠 中危

**整改要点**:
- 增强 sessions 表（添加设备信息、位置等）
- 实现"查看活跃会话"功能
- 实现"登出所有设备"功能
- 添加可疑登录检测（新设备/新地理位置）
- 发送登录通知邮件

**实施时间**: 3周内

---

### 8. 附件安全检查不足

**问题描述**:
- 缺少文件类型验证（基于 magic number）
- 无病毒扫描机制
- 附件下载未签名
- 缺少内容类型验证

**风险等级**: 🟠 中危

**整改要点**:
- 实现基于 Magic Number 的文件类型验证
- 文件大小限制（如 10MB）
- 实施文件类型白名单
- 附件下载 URL 签名和过期机制
- 考虑集成病毒扫描 API
- 强制 Content-Disposition: attachment

**实施时间**: 3周内

---

### 9. 速率限制粒度优化

**问题描述**:
- 当前速率限制较为粗糙
- 需要基于用户等级的分层限制
- 敏感操作需要特殊限制

**风险等级**: 🟠 中危

**整改要点**:
- 实现分层速率限制（guest/free/premium）
- 为敏感操作设置特殊限制
- 添加每小时限制（除了每分钟和每天）
- 优化限制粒度和阈值
- 添加速率限制监控和告警

**实施时间**: 2周内

---

### 10. 审计日志增强

**问题描述**:
- 当前审计日志信息不够详细
- 缺少数据变更追踪（before/after）
- 未实现日志不可篡改性
- 缺少请求 ID 追踪

**风险等级**: 🟠 中危

**整改要点**:
- 增强审计日志表结构（before/after 值）
- 实现日志链（防篡改）
- 记录所有敏感操作
- 添加请求 ID 追踪
- 实现日志完整性验证
- 定期导出到外部系统

**实施时间**: 3周内

## 🟡 低危问题（长期优化）

### 11. 密码强度策略优化

**当前要求**: 8位 + 大小写 + 数字

**改进建议**:
- 最低 12 位（建议 16 位）
- 必须包含特殊字符
- 检查常见密码字典
- 集成 Have I Been Pwned API
- 显示实时密码强度评分

**实施时间**: 1个月

---

### 12. 多因素认证 (MFA)

**状态**: 未实现

**改进建议**:
- 实现 TOTP 支持（Google Authenticator）
- 用户可选启用 2FA
- 生成备用恢复码
- 登录时验证 MFA token
- 支持 SMS 验证（可选）

**实施时间**: 2个月

### 13. API 密钥轮换提醒

**问题**: 无过期提醒机制

**改进建议**:

- API 密钥过期前 7 天发送邮件提醒

- 支持临时禁用密钥
- 记录详细使用统计

**实施时间**: 1-2个月

**问题**: 依赖手动清理

**改进建议**:
- 用户可自定义邮件保留时间（7/14/30天）
- 软删除机制保留 7 天
- 附件单独计算存储配额

- 添加 X-Frame-Options

- 添加速率限制

### 18. 邮件头验证

- 记录验证结果

### 19. API 响应数据脱敏

- 错误消息不暴露内部细节
